# 이음(理音) 역할 연동·프로세스 완결성 점검 보고서

- 점검일: 2026-07-14
- 대상: 후원자 · 협의체 위원 · 운영자 모바일 클릭형 프로토타입
- 대상 URL: https://kevinsaem.github.io/ieum-click-prototype/
- 목적: 세 역할의 업무가 앱 안에서 실제 운영 가능한 수준으로 인수인계되는지 점검

## 1. 총평

현재 프로토타입은 **핵심 정상 경로(Happy Path)의 방향은 적절하다.**

```text
기존 제공 가능 후원
→ 협의체 위원 케이스 등록
→ 자동 추천 또는 수동 연결
→ 매칭 신청
→ 운영 검토
→ 서로 다른 승인자 2인 공동 승인
→ 재고 차감
→ 배송 대기
→ 배송 중
→ 완료
```

위 흐름에서 매칭 신청은 후원자와 운영자 목록에 나타나고, 최종 승인 뒤 재고와 케이스 상태가 함께 변경되며, 배송 완료 시 매칭과 케이스가 모두 완료된다. 따라서 서비스의 핵심 개념과 역할 분담은 잘 잡혀 있다.

그러나 **새로운 후원과 새로운 케이스가 정상 경로에 진입하는 앞단**, **후원자의 실제 제공 확약**, **수정·취소·부분 완료 같은 예외 처리**, **완료 이후 기록·통계 연동**, **영구 저장과 권한 통제**에는 중요한 단절이 있다. 현재 상태는 제품 검증용 프로토타입으로는 의미가 있지만, 실제 운영 가능한 앱으로 보기에는 추가 개발이 필요하다.

## 2. 역할별 정상 연결 상태

| 단계 | 후원자 | 협의체 위원 | 운영자 | 현재 연동 평가 |
|---|---|---|---|---|
| 후원 등록 | 물품·식사·서비스와 사진 등록 | 등록된 제공 가능 후원 확인 | 승인 대기 후원 목록 확인 | 부분 연결 |
| 케이스 등록 | 식별정보 열람 없음 | 비식별 케이스와 사진 등록 | 신규 케이스 전용 검토 없음 | 단절 |
| 자동 추천 | 기존 제공 가능 후원이 추천 후보가 됨 | 카테고리·수량·기한 기준 추천 선택 | 아직 개입하지 않음 | 연결됨 |
| 수동 연결 | 제공 가능 후원이 카탈로그에 노출 | 후원 상세 확인 후 미매칭 케이스 선택 | 아직 개입하지 않음 | 연결됨 |
| 매칭 신청 | 자신의 후원과 연결된 요청 확인 | 수량·일정·방식·담당자·사유 제출 | 검토함에서 제안 확인 | 연결됨 |
| 제공 확약 | `제공 가능 확인` 버튼 존재 | 확인 여부를 볼 수 없음 | 확인 여부와 무관하게 승인 가능 | 단절 |
| 공동 승인 | 승인 결과를 간접 확인 | 나의 활동에서 상태 확인 | 서로 다른 승인자 2인 검증 | 프로토타입 수준 연결 |
| 재고 예약 | 남은 수량 확인 | 승인 전 확정되지 않음 | 최종 승인 시 수량 차감 | 연결됨 |
| 배송 | 매칭 상세에서 제한적으로 확인 | 케이스 진행 상태 확인 | 배송 대기→배송 중→완료 처리 | 부분 연결 |
| 완료 기록 | 정적 예시 기록만 표시 | 동적 활동 상태 확인 | 케이스·매칭 완료 상태 변경 | 부분 연결 |
| 보고 | 별도 기능 없음 | 별도 기능 없음 | 정적 월간 보고서 | 단절 |

## 3. 실제 브라우저 검증 결과

### 3.1 정상적으로 연결된 핵심 경로

새 식사 케이스를 등록하고 추천 자원을 선택해 매칭을 제출한 결과:

- 매칭 상태: `운영 검토 대기`
- 후원자 매칭 목록: 신규 요청 표시
- 운영자 검토함: 동일 비식별 케이스 표시
- 첫 번째 승인: `공동 승인 1/2`, 재고 유지
- 두 번째 승인: `배송 대기`, 재고 `20세트 → 18세트`
- 배송 시작: `배송 중`
- 전달 완료: 매칭 `완료`, 케이스 `완료`

이 구간은 세 역할의 핵심 데이터가 같은 매칭을 기준으로 움직인다.

### 3.2 후원 등록 인수인계 단절

후원자가 `통합점검 반찬 세트`를 등록한 결과:

- 후원 상태: `승인 대기`
- 운영자 `후원 등록 검토` 목록: 정상 표시
- 운영자 목록의 승인·반려·상세 검토 버튼: 없음
- 목록 클릭 대상: 없음

즉, 등록 정보는 운영자에게 전달되지만 운영자가 승인하여 `제공 가능` 상태로 전환할 수 없다. 신규 후원은 연결 카탈로그에 진입하지 못한다.

### 3.3 케이스 검토 단계 누락

협의체 위원이 새 케이스를 등록한 결과:

- 등록 직후 상태: `자원 탐색`
- 자동 추천: 2건 표시
- 운영자 신규 케이스 검토함: 해당 케이스 미표시

현재는 비식별·동의·필요성·중복·긴급도 검증을 운영자가 수행하기 전에 자동 추천과 매칭 신청으로 진입할 수 있다.

### 3.4 후원자 제공 확인이 상태에 반영되지 않음

후원자 매칭 상세에서 `제공 가능 확인`을 누른 결과:

- 안내 토스트: 표시됨
- 매칭 상태: 변경 없음
- 매칭 데이터: 변경 없음
- 후원자 확인자·확인일시: 저장되지 않음

운영자는 후원자의 실제 제공 확약 없이도 공동 승인을 완료할 수 있었다.

### 3.5 후원자 매칭 상세가 실제 매칭 데이터와 불일치

실제 완료된 신규 매칭 데이터:

- 후원: `한 끼 도시락 2세트`
- 상태: `완료`

후원자 상세 화면:

- `도시락 7세트`
- `협의체 공동 승인 검토 대기`
- `전달 일정은 승인 후 정해짐`

목록은 동적 데이터지만 상세 화면 일부가 고정 예시이므로 실제 상태와 다른 안내를 보여준다.

### 3.6 수정 요청 이후 보완 경로 누락

운영자가 매칭에 수정 요청한 결과:

- 매칭: `수정 요청`
- 후원 자원: `연결 대기`로 복귀
- 케이스 상태: 기존 `매칭 검토` 유지
- 위원 활동 화면의 수정·보완·재제출 버튼: 없음

위원은 기존 제안을 수정하는 대신 사실상 새로운 매칭을 다시 만들어야 하며, 수정 이력이 하나의 업무 건으로 연결되지 않는다.

### 3.7 완료 이후 기록·보고 단절

신규 매칭을 배송 완료한 결과:

- 매칭과 케이스 상태: 정상적으로 `완료`
- 후원자 나눔 기록: 신규 완료 건 미표시
- 운영 보고서: `신규 케이스 8건 · 완료 매칭 14건 · 참여 가게 7곳` 고정 표시

완료 데이터가 기록과 집계 화면까지 이어지지 않는다.

### 3.8 예외 배송 흐름 부재

배송 중 운영자 화면에서 제공되는 작업:

- `전달 완료 기록`만 제공

없는 작업:

- 부분 전달
- 배송 실패
- 일정 변경
- 수령 거부
- 취소
- 재배송
- 잔여 수량 재매칭

### 3.9 영구 저장 부재

통합 점검 중 생성한 후원·케이스·매칭은 새로고침 후 모두 사라졌다.

- 초기 후원: 7건으로 복귀
- 초기 케이스: 2건으로 복귀
- 초기 매칭: 1건으로 복귀
- 점검 중 생성한 후원·케이스: 없음

이는 정적 프로토타입의 의도된 한계지만 실제 앱 개발에서는 최우선 기반 과제다.

## 4. 누락 사항과 심각도

### P0 · 실제 운영을 막는 핵심 단절

#### GAP-01. 후원 등록 승인·반려·보완 기능 없음

**영향:** 신규 후원이 영원히 `승인 대기`에 머물며 연결 목록에 진입하지 못한다.

필요 기능:

- 후원 상세 검토
- 사진·수량·기한·전달 방식 확인
- 승인
- 보완 요청
- 반려
- 승인자·처리 사유·처리일시 저장
- 승인 시 `제공 가능 / 연결 대기` 전환

#### GAP-02. 신규 케이스 운영 검토 없음

**영향:** 동의 근거, 비식별 여부, 중복 케이스, 긴급도, 지원 적정성을 검토하지 않고 매칭 신청할 수 있다.

필요 기능:

- `케이스 검토 대기`
- 운영자 케이스 검토함
- 승인·보완 요청·종결
- 승인 후에만 `자원 탐색`과 자동 추천 활성화
- 긴급 케이스의 신속 검토 규칙

#### GAP-03. 영구 저장·인증·권한 통제 없음

**영향:** 새로고침 시 업무가 사라지고, 실제 사용자별 데이터 범위와 승인 권한을 보장할 수 없다.

필요 기반:

- API와 데이터베이스
- 로그인과 조직 소속
- 역할 기반 접근 제어(RBAC)
- 후원자·위원·운영자 계정 분리
- 서버 기준 상태 전이 검증
- 감사 로그
- 사진 객체 저장소

### P1 · 세 역할의 인수인계를 불완전하게 만드는 문제

#### GAP-04. 후원자의 제공 확약이 실제 승인 조건이 아님

권장 상태:

```text
매칭 신청
→ 후원자 제공 가능 확인
→ 운영 검토
→ 공동 승인
```

또는 긴급 상황에서 운영 검토를 먼저 할 수 있더라도, 최종 재고 예약 전에는 후원자 확인이 필수여야 한다.

저장 항목:

- donor_confirmed_at
- donor_confirmed_by
- confirmed_quantity
- pickup_window
- unavailable_reason

#### GAP-05. 후원자 매칭 상세가 고정 데이터

모든 제목·수량·상태·일정·타임라인을 선택된 매칭 객체에서 렌더링해야 한다.

#### GAP-06. 수정 요청→보완→재제출 흐름 없음

권장 흐름:

```text
운영 수정 요청
→ 위원에게 사유와 수정 항목 표시
→ 기존 제안 편집
→ 재제출
→ 동일 매칭 ID로 재검토
```

케이스와 후원 자원의 상태도 함께 원복하거나 잠금 해제해야 한다.

#### GAP-07. 완료 데이터가 기록과 보고에 연결되지 않음

필요 연동:

- 후원자 완료 기록
- 위원 활동 실적
- 운영자 완료 목록
- 월간 집계
- 잔여 수량
- 참여 조직 실적
- 비식별 PDF 보고서

#### GAP-08. 배송 예외 흐름 없음

필요 상태:

- 배송 일정 변경
- 픽업 준비 완료
- 픽업 완료
- 부분 전달
- 전달 실패
- 재배송 대기
- 취소
- 잔여 수량 재매칭

### P2 · 운영 안정성과 확장성을 위한 보완

#### GAP-09. 재고 동시성·예약 원장 부족

현재는 하나의 자원이 검토 중일 때 목록에서 빠져 중복 신청을 완화하지만, 실제 서버 트랜잭션과 수량 예약이 아니다.

필요 모델:

- total_quantity
- available_quantity
- pending_quantity
- reserved_quantity
- delivered_quantity
- inventory_transactions
- 예약 만료와 해제

#### GAP-10. 알림과 업무 기한 없음

필요 알림:

- 후원 등록 검토 요청
- 케이스 보완 요청
- 매칭 제안 도착
- 후원자 제공 확인 요청
- 공동 승인 요청
- 배송 예정·지연
- 완료·취소

필요 운영 정보:

- 담당자
- 처리 기한
- 지연 상태
- 읽음 여부
- 재알림

#### GAP-11. 실제 공동 승인 신뢰성 부족

현재는 화면에서 승인자 이름을 선택한다. 실제 앱에서는 로그인한 사용자 본인만 승인할 수 있어야 한다.

필요 조건:

- 서버가 로그인 사용자 ID로 승인자 기록
- 동일 사용자 중복 승인 차단
- 역할·조직·이해충돌 검증
- 승인 취소와 사유
- 모든 변경의 감사 로그

#### GAP-12. 개인정보·사진 검수 자동화 부족

현재 안내 문구는 적절하지만 실제 운영에서는 다음이 필요하다.

- 사진 EXIF 제거
- 얼굴·문서·전화번호·주소 포함 여부 검수
- 원본 접근 권한
- 보관 기간과 자동 삭제
- 다운로드 제한
- 식별정보 별도 암호화 저장

## 5. 권장 통합 상태 모델

### 5.1 후원 자원

```text
임시 저장
→ 운영 검토 대기
→ 보완 요청 / 반려
→ 제공 가능(연결 대기)
→ 매칭 검토 중
→ 예약
→ 전달 중
→ 일부 잔여 / 소진
→ 종료
```

### 5.2 케이스

```text
임시 저장
→ 케이스 검토 대기
→ 보완 요청 / 종결
→ 승인
→ 자원 탐색
→ 매칭 신청
→ 운영·공동 승인
→ 전달 일정 확정
→ 전달 중
→ 부분 완료 / 완료 / 취소
```

### 5.3 매칭

```text
작성 중
```

## 6. 권장 개발 순서

### 1단계 · 업무 진입점 완결

1. 운영자 후원 승인·반려
2. 운영자 케이스 승인·보완
3. 후원자 제공 확약 저장
4. 위원 수정·재제출
5. 각 상태의 담당자와 처리일시 저장

### 2단계 · 전달·완료 완결

1. 픽업 준비·픽업 완료

5. 알림과 지연 업무 표시

1. 데이터베이스와 API
2. 로그인·조직·RBAC
3. 서버 상태 전이와 재고 트랜잭션
4. 사진 저장·검수·삭제 정책
5. 감사 로그
6. 자동 테스트와 운영 모니터링

## 7. 최종 결론

이음의 세 역할은 현재 **매칭 신청 이후의 정상 경로에서는 유기적으로 연결된다.** 특히 비식별 케이스, 자동 추천, 위원 제안, 운영 검토, 서로 다른 승인자 2인, 승인 후 재고 차감, 배송 완료까지의 제품 방향은 타당하다.

하지만 실제 앱의 프로세스 완결성을 위해서는 다음 다섯 가지를 우선 보완해야 한다.

1. 후원 등록 승인
2. 케이스 등록 검토
3. 후원자 제공 확약
4. 수정·예외·부분 완료 흐름
5. 완료 기록·통계·영구 저장

이 다섯 구간을 연결하면 세 역할이 단순히 화면에서 나뉘는 수준을 넘어, 하나의 업무 건을 처음부터 끝까지 공동 처리하는 실제 운영 앱 구조에 가까워진다.
